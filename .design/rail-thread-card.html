Thread HoverCard · 重设计

子区的「头像」其实是借父群的头像。当前卡片把同一张脸同时显示在顶部 parent strip（16px）和主体头像位（36px）位置——视觉冗余、抢焦点、用户认知负担。本稿给 4 个去重方案，推荐 Breadcrumb 单层。

容器宽 256–280px · 主题色 hash(thread.name) · 左侧浮出 · 纯只读

01 · 问题诊断

当前实现 → 重复脸

截图实拍：父群「虾饼鞋匠」的卡通脸出现两次。一张做 16px 父群标识、一张做 36px 主体头像。两张完全相同的图，意义重复，更糟的是抢走了「充值」这个子区名的视觉焦点。

问题清单：① 两张同图视觉冗余 ② 子区名"充值"字号被压制（14px）③ 头像和子区名同行排版导致主体留白消失 ④ parent strip 占了整整一行高度但只为传一句话

设计目标

  • G1 主头像不复用父群图 —— 子区有自己的身份，用色块+#符号建立独立视觉
  • G2 子区名突出 —— 字号 ≥ 16px，作为视觉绝对主角
  • G3 父群退到次要 —— 上下文存在，但不能跟子区平起平坐
  • G4 视觉与 rail 角标连贯 —— rail 右下 # 角标的颜色（hash 子区名）延伸到卡片主头像，强化「同色 = 同子区」

关键洞察：子区是个「话题」而不是「群」。话题的图形语言是 #，群的图形语言是头像。当前实现错把子区当群处理。

04 · 与 Rail 的颜色呼应

rail 角标色 → 卡片主头像色

用户在 rail 上看到的子区右下角 # 角标颜色（hash 子区名），在 hover 卡里被放大成主头像背景色。同一子区 rail 和 hover 视觉绑定。

// Rail 角标

05 · 实施清单

改动锚点（B 方案）

  1. P0-1 · src/components/octo/RailHoverCard.tsx：把当前的 parent-strip + grid head 重构成 「crumb + body」 两段式。
  2. P0-2 · 子区分支单独走一套布局；私聊/群聊保持现状不变。用 isThread 切。
  3. P0-3 · 主头像用 getThreadHueColor(name) 作为背景色，居中 # 符号；font-size 22px，icon 容器 40×40。
  4. P0-4 · Crumb 用 inline-flex pill 样式，bg-primary/8 背景、border-primary/12 描边、rounded-full。不再独占整行。
  5. P1-1 · 子区名升级到 16px / font-weight 700，比群聊卡的 14px 更突出（话题感）。
  6. P1-2 · @me 时让 --thread-hue 临时变 #F97316（可选；先做保守版只加 pill）。
  7. P2 · 父群头像加载失败 fallback 到 getInitials(parentName, 1)+getTitleColor 色块。
